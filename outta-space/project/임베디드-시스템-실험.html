구현해야 했던 것은 무엇인가? - input으로 어떤 음성 데이터가 들어온다. 이 음성 데이터는 spcetrogram으로 변환되어 flatten되고, 이것이 16개 batch되어 들어간다. (미리 타일링 됨)

이런 걸 4번 반복해서 사용해야 한다. 그니까 우리가 구현해야할 건...

바로 이것이다. 여기서 BRAM을 제외한 전부를 구현하면 된다. PE는 int8 input을 받아 곱해서(int16) 더한걸(int32) 출력하는 구조이다. Post Processor는 relu와 requantization, rounding, saturating을 적용하는 구조이다. Sequencer도 구현해야 한다. Skewer는 PE array에 input을 공급하는 역할을 한다. 시퀀서가 잘 이해가 안가네…

보고서에 들어가야 하는 내용은 무엇인가? - 시스톨릭 어레이의 작동 원리, 타일링에 대해 기재. 어떻게 하드웨어에 맞춰서 설계했는지? material description에 대해 기재. 그니까 BRAM에 서 어떻게 데이터를 읽고 쓰는지를 설명하면 될듯. 코드 설명 부분은 당연히 포함이다. 그리고 어려웠던 점... 은 codex를 읽고 알아보자. + 시퀀서의 3중 for 문을 어떻게 구현했는지.

discussion 부분 ​

  • 어려웠던 점)

가장 크게 막혔던 부분은 RTL 구조를 명세에서 실제 동작 가능한 데이터 흐름으로 바꾸는 것이었습니다. 단순히 PE 하나를 만드는 것과 달리, 16x16 systolic array에서는 입력 A/B가 어느 cycle에 어느 PE에 도착해야 하는지, 누산값이 언제 유효해지는지, flush를 몇 cycle 해야 하는지 맞추는 게 까다로웠습니다. 중간에 C11 expected 50, got 18 같은 오류가 난 것도 이 타이밍 정렬 문제가 원인이었습니다.

두 번째는 BRAM 주소와 128-bit packing 방식이었습니다. AXI에서는 32-bit 단위로 접근하지만 내부 BRAM은 128-bit word 단위라서, addr[17:4]가 BRAM row 주소가 되고 addr[3:2]가 32-bit lane이 되는 구조였습니다. 그래서 0x2880 같은 RTL 주소가 Vitis에서는 0x28800 byte offset이 되는 식으로 변환해야 했고, 이 부분이 헷갈리기 쉬웠습니다.

세 번째는 layer를 이어 붙이는 CONTROL FSM이었습니다. 처음에는 layer1 tile만 검증했고, 이후 layer2, layer3, layer4를 이어 붙이면서 input/output base address, weight base address, k-tile 수, out-tile 수, scaler를 layer별로 바꾸도록 확장했습니다. 여기서 실수하면 이전 layer output을 다음 layer input으로 제대로 못 읽거나, 결과를 엉뚱한 BRAM 위치에 쓰게 됩니다.

네 번째는 post-processing timing closure였습니다. 처음에는 ReLU, 32x32 multiply, rounding, saturation을 한 클럭에 처리해서 implementation에서 WNS가 약 -9ns까지 깨졌습니다. 이건 기능 시뮬레이션에서는 통과해도 FPGA에서는 동작 보장이 안 되는 문제였습니다. 그래서 POST_PROCESSOR를 여러 stage로 pipeline화해서 timing을 통과시켰고, 이후 DSP pipeline 경고도 줄이기 위해 곱셈 뒤 register stage를 추가했습니다.
