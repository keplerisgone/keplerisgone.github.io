정보처리기사 시나공 기출 요약 ​

1. 요구사항 확인 ​

  • 애자일: 고객의 변화하는 요구사항, 고객과의 소통 중심, 사이클 중심, 절차, 문서, 계획 << 소통, 협업, 변화 대응
    • Scrum: 스프린트 단위로 개발하고 스크럼 마스터, 제품 책임자, 개발팀이 협업하는 방법론
    • XP: 짧은 개발 주기와 지속적 통합, 테스트, 리팩토링을 강조하는 방법론
    • Kanban: 작업 흐름을 시각화하고 진행 중 작업량을 제한해 효율을 높이는 방법론
    • Crystal: 프로젝트 규모와 위험도에 따라 프로세스를 조정하는 경량 방법론
    • FDD: 기능 단위로 계획, 설계, 구현을 반복하는 방법론
    • ASD: 변화 수용과 학습을 중시하는 적응형 개발 방법론
    • DSDM: 정해진 시간과 비용 안에서 우선순위에 따라 점진적으로 개발하는 방법론
  • 요구사항 분류
    • 비기능적 요구사항: 성능, 보안, 품질, 안전 등에 대한 요구사항
    • 기능적 요구사항: 세스템의 기능, 제어 연산, 기술에 대한 요구사항
  • UML: 소프트웨어 공학에서 사용되는 표준화된 범용 모델링 언어
    • 사물, 관계, 다이어그램으로 구성
    • 사물: 구조(structural), 행동(behavioral), 그룹(Grouping), 주해(annotation)
      • 구조 사물은 다시 Class, Interface, Usecase, Node 등으로 나뉨
    • 관계: Assosiation(연관), Aggregation(집합), Composition(포함), Generalization(일반화), Dependency(의존), Realization(실체화) 관계로 나눌 수 있다.
    • 다이어그램:
      • 구조적 다이어그램: 클래스 다이어그램, 객체 다이어그램, 컴포넌트 다이어그램, 배치 다이어그램, 복합체 다이어그램, 패키지 다이어그램
      • 행위적 다이어그램: 유스케이스 다이어그램, 상태 다이어그램, 활동 다이어그램, 절차 다이어그램, 통신 다이어그램, 타이밍 다이어그램, 상호작용 개요 다이어그램
  • 비용 산정 기법:
    • 하향식 비용 산정 기법: 비과학적인 기법, 전문가 측정 기법, 델파이 측정 기법
    • 상향식 비용 산정 기법:
      • LOC: 낙관치, 기대치, 비관치 등을 기반으로 전체 비용을 산정하는 법
      • 예측치:
        • (낙관치+4x기대치+비관치)/6
      • line of code를 사용해 계산한다. - LOC/인당 월평균 생산 코드 라인/투입 인원
      • 단계별 노력 기법도 있다.
    • 수학적 산정 기법
      • COCOMO:
        • 보헴이 제안한 LOC 기반 비용 산정 기법
        • 규모에 따라 Basic, Intermediate, Detailed로 나뉨.
        • Basic은 KDSI에 따라 Organic - Semi-Detached - Embedded로 나뉨.

2. 데이터 입출력 구현 ​

  • 데이터베이스 시스템
    • 데이터의 종류:
      • 공유 데이터(shered data): 공통으로 필요로 하는 데이터
      • 통합 데이터(integrated data): 불필요한 데이터를 제거하고 중복이 최소화된 데이터
      • 저장 데이터(stored data): 시스템이 접근할 수 있는 곳에 저장된 데이터
      • 운영 데이터(operational data): 조직의 목적을 위해 지속적으로 필요한 데이터
    • DB 시스템 특징: 실시간 접근성, 계속적인 진화, 동시 공유, 내용에 의한 참조, 데이터의 논리적 독립성
  • 스키마:
    • 데이터베이스의 자료 구조와 표현 방법, 관계 등을 정의한 제약조건이다.
    • 외부, 개념, 내부 스키마로 나뉜다.
    • 외부 스키마: 특정 사용자의 입장에서 필요한 데이터베이스 구조를 정의 (제일 바깥의 것)
    • 개념 스키마: 조직의 입장을 고려한 것
    • 내부 스키마: 물리적인 저장장치 입장을 고려한 것
  • 데이터베이스 설계 단계:
    • 개념적 설계: 현실 세계의 요구사항을 개념적 데이터 모델로 표현
    • 논리적 설계: 개념적 모델을 DBMS가 처리할 수 있는 논리적 구조로 변환
    • 물리적 설계: 저장 구조, 인덱스, 접근 경로 등 물리적 구현 방식을 설계
  • 데이터 모델링의 구조:
    • 현실 세계의 개념적인 기업의 정보 구조를 디지털 세계의 논리적인 데이터 모델로 명확하고 체계적으로 변환해 문서화하는 기법을 말한다.
    • 표시 요소: (논리적) 구조, 연산, 제약 조건
    • 구성 요소: 개체, 속성, 관계
  • Cardinality: 튜플에서의 총 개체 수
  • Degree: 튜플에서의 총 속성 수
  • E-R 다이어그램 (ERD):
    • Peter chan이 최초로 제안, 개체, 관계, 속성으로 이루어짐.
    • 사각형이 개체, 원형은 속성, 마름모는 관계 타입
  • 키 (Key):
    • 데이터를 분류, 정렬, 검색할 때의 기준이 되는 속성을 의미
    • 후보키: 유일성과 최소성을 만족
    • 기본키: 후보키들 중 튜플의 식별을 위해 지정된 속성
    • 슈퍼키: 유일성을 만족, 최소성은 불만족
  • 정규화
    • 데이터 무결성을 유지하기 위해 중복성을 최소화하고 정보의 일관성을 보장하기 위한 개념
    • 이상 현상을 제거하기 위함:
      • 삽입 이상(관련 없는 정보도 함께 삽입)
      • 갱신 이상(데이터의 불일치가 발생하는 현상)
      • 삭제 이상(데이터를 삭제할 때 관련된 정보도 함께 삭제해야 하는 현상)
    • 도부이결다조: 도메인이 원자값(1NF), 부분 함수 종속 제거(2NF), 이행 함수 종속 제거(3NF), 결정자가 후보키(BCNF), 다치 종속 제거(4NF), 조인 종속 제거(5NF)
    • 반정규화: 일부러 중복, 분리 등을 수행하는 것
    • 비정규화: 정규화가 진행되지 않은 상태
  • 데이터 종속성:
    • 함수 종속: 한 속성이 다른 속성을 유일하게 식별할 수 있는 상태
      • 학번(결정자)은 중복되지 않기 때문에 학생의 다른 속성(종속자)들을 유일하게 식별한다.
    • 완전 함수 종속 (Full Functional Dependency): 종속자가 모든 결정자에 종속되는 경우
    • 부분 함수 종속 (Partial Functional Dependency): 종속자가 일부 결정자에 종속되는 경우
    • 이행적 함수 종속 (Transitive Dependency): 종속자 C의 결정자가 여러 개인 경우
  • 파일의 구조:
    • 순차 파일: 레코드를 논리적 순서대로 연속 저장하는 구조
    • 인덱스 파일: 인덱스를 이용해 원하는 레코드에 빠르게 접근하는 구조
    • 해싱 파일: 해시 함수를 이용해 레코드의 저장 위치를 계산하는 구조

3. 통합 구현 ​

  • EAI:
    • Point-to-Point: 애플리케이션끼리 1:1로 직접 연결하는 방식
    • Hub & Spoke: 중앙 허브를 통해 각 애플리케이션을 연결하는 방식
    • Message Bus: 공통 메시지 버스를 통해 데이터를 교환하는 방식
    • Hybrid: Hub & Spoke와 Message Bus 등을 혼합한 방식
  • 웹 서비스 연동:
    • SOAP: XML 기반 메시지로 웹 서비스 간 정보를 교환하는 프로토콜
    • UDDI: 웹 서비스를 등록하고 검색하기 위한 저장소
    • WSDL: 웹 서비스의 인터페이스와 호출 방법을 XML로 기술하는 언어

4. 서버 프로그램 구현 ​

  • 리팩토링: 코드 품질 향상이 목적, 외부 기능의 변화 없이 내부 구조를 개선하는 작업 (이해하기 쉽게 만드는 것)
  • 럼바우:
    • 객체 모델링: 시스템의 객체와 객체 간 관계를 표현
    • 동적 모델링: 상태 변화와 이벤트 흐름을 표현
    • 기능 모델링: 자료 흐름과 처리 기능을 표현
  • 객체 지향의 SOLID
    • 단일 책임 원칙(SRP): 클래스는 하나의 책임만 가져야 함
    • 개방-폐쇄 원칙(OCP): 확장에는 열려 있고 변경에는 닫혀 있어야 함
    • 리스코프 치환 원칙(LSP): 하위 타입은 상위 타입을 대체할 수 있어야 함
    • 인터페이스 분리 원칙(ISP): 클라이언트는 사용하지 않는 인터페이스에 의존하지 않아야 함
    • 의존성 역전 원칙(DIP): 구체 클래스보다 추상화에 의존해야 함
  • 결합도: 줄여야 하는 것 (Coupling)
    • Data: 필요한 데이터만 매개변수로 전달하는 낮은 결합도
    • Stamp: 자료 구조를 통째로 전달해 일부만 사용하는 결합도
    • Control: 제어 플래그를 전달해 다른 모듈의 흐름을 제어하는 결합도
    • External: 외부 장치나 외부 데이터 형식을 공유하는 결합도
    • Common: 공통 데이터 영역을 여러 모듈이 공유하는 결합도
  • 응집도: 높여야 하는 것 (Cohesion)
    • Functional: 하나의 기능을 수행하는 가장 높은 응집도
    • Sequential: 한 활동의 출력이 다음 활동의 입력이 되는 응집도
    • Communication: 같은 입력이나 출력을 사용하는 활동들이 모인 응집도
    • Procedural: 정해진 절차에 따라 수행되는 활동들이 모인 응집도
    • Temporal: 특정 시간에 함께 수행되는 활동들이 모인 응집도
    • Logical: 유사한 성격의 활동들이 논리적으로 묶인 응집도
    • Coincidental: 관련 없는 기능들이 우연히 모인 가장 낮은 응집도
  • 디자인 패턴
    • 생성 패턴: 객체 생성 방식을 캡슐화하거나 생성 과정을 분리한다.
      • Abstract Factory: 관련 객체군을 구체 클래스 지정 없이 생성
      • Builder: 복잡한 객체의 생성 과정과 표현을 분리
      • Factory Method: 객체 생성을 서브클래스에 위임
      • Prototype: 기존 객체를 복제해 새 객체 생성
      • Singleton: 클래스의 인스턴스를 하나만 생성해 공유
    • 구조 패턴: 클래스나 객체를 조합해 더 큰 구조를 만든다.
      • Adapter: 호환되지 않는 인터페이스를 변환
      • Bridge: 추상부와 구현부를 분리해 독립적으로 확장
      • Composite: 객체를 트리 구조로 구성해 단일 객체와 복합 객체를 동일하게 처리
      • Decorator: 객체에 기능을 동적으로 추가
      • Facade: 복잡한 서브시스템에 단순한 통합 인터페이스 제공
      • Flyweight: 공통 데이터를 공유해 메모리 사용 절감
      • Proxy: 실제 객체 접근을 대리하거나 제어
    • 행위 패턴: 객체 간 역할 분담과 상호작용 방식을 정의한다.
      • Chain of Responsibility: 요청을 처리할 객체를 사슬처럼 연결
      • Command: 요청을 객체로 캡슐화
      • Interpreter: 언어 문법을 클래스로 표현하고 해석
      • Iterator: 컬렉션 내부 구조를 노출하지 않고 순차 접근
      • Mediator: 객체 간 상호작용을 중재 객체로 집중
      • Memento: 객체 상태를 저장하고 복원
      • Observer: 한 객체의 상태 변화 시 의존 객체에 자동 통지
      • State: 객체 상태에 따라 행위를 변경
      • Strategy: 알고리즘을 캡슐화해 교체 가능하게 함
      • Template Method: 상위 클래스에서 알고리즘 골격을 정의하고 일부 단계는 하위 클래스에서 구현
      • Visitor: 객체 구조는 유지한 채 새로운 연산을 추가
  • IPC: 프로세스 통신
    • 공유 메모리: 여러 프로세스가 같은 메모리 영역을 공유
    • 메시지 전달: 프로세스 간 메시지를 주고받아 통신
    • 소켓: 네트워크를 통해 프로세스 간 데이터를 송수신
    • 세마포어: 공유 자원의 접근 순서를 제어
    • 파이프/네임드 파이프: 단방향 또는 이름 있는 통신 경로를 제공
    • 메시지 큐잉: 메시지를 큐에 저장하고 순차적으로 전달

5. 인터페이스 구현 ​

  • 데이터 표현 방식:
    • XML: 태그 기반으로 데이터를 계층 구조로 표현하는 마크업 언어
    • JSON: 키-값 쌍과 배열로 데이터를 표현하는 경량 데이터 형식
    • YAML: 들여쓰기를 이용해 사람이 읽기 쉬운 형태로 데이터를 표현하는 형식
    • AJAX: 비동기 통신으로 웹 페이지 일부를 갱신하는 기술

6. 화면 설계 ​

  • UI 설계 원칙
    • 직관성: 사용자가 쉽게 이해하고 사용할 수 있어야 함
    • 유효성: 사용자의 목적을 정확하고 완전하게 달성할 수 있어야 함
    • 학습성: 사용자가 쉽게 배우고 익힐 수 있어야 함
    • 유연성: 사용자의 요구와 환경 변화에 대응할 수 있어야 함

7. 애플리케이션 테스트 관리 ​

  • 소프트웨어 테스트 진행
    • 결함 집중: 결함은 특정 모듈에 집중되는 경향이 있음
    • 낚시의 법칙: 경험과 직관을 활용해 결함 가능성이 높은 부분을 찾음
    • 파레토의 법칙: 전체 결함의 상당수가 소수 모듈에서 발생함
    • 살충제 패러독스: 같은 테스트를 반복하면 새로운 결함을 발견하기 어려움
    • 오류-부재의 궤변: 결함이 없어도 요구사항을 만족하지 못하면 품질이 높다고 할 수 없음
    • 테스트 항목: 테스트할 대상 기능이나 모듈
    • 테스트 조건: 테스트가 확인해야 하는 상황이나 기준
    • 테스트 데이터: 테스트 수행에 필요한 입력값
  • 단위 테스트 도구:
    • JUnit: Java 프로그램의 단위 테스트를 지원하는 프레임워크
    • CppUnit: C++ 프로그램의 단위 테스트를 지원하는 프레임워크
  • 테스트 유형:
    • 정적 테스트: 소스 코드만을 보고 테스트하는 것 (워크스루, 인스펙션, 동료 검사 등)
    • 동적 테스트: 프로그램의 실행 화면을 보면서 테스트 하는 것 (화이트박스 테스트, 블랙박스 테스트)
  • 테스트 커버리지:
    • 테스트 범위를 측정하는 기준
    • 기능 기반 커버리지: 요구 기능이 테스트된 정도를 측정
    • 라인 커버리지: 전체 코드 라인 중 실행된 라인의 비율을 측정
    • 코드 커버리지: 구문 커버리지(모든 구문 테스트), 결정 커버리지(참과 거짓), 조건 커버리지(결정문 내부 조건식의 결과), 조건/결정 커버리지, 다중 조건 커버리지
  • 화이트박스 테스트 기법
    • 기초 경로(Basic Path) 테스트: 프로그램의 독립 실행 경로를 테스트
    • 조건 검사: 조건문의 참/거짓과 조합을 테스트
    • 루프 검사: 반복문의 반복 횟수와 경계 조건을 테스트
    • 데이터 흐름 검사: 변수 정의와 사용 흐름을 추적해 테스트
  • 블랙박스 테스트 기법
    • 동등 분할(Equivalence Partitioning): 입력값을 동등한 그룹으로 나누어 대표값을 테스트
    • 원인 효과-그래프(Cause-Effect Graphing): 입력 조건과 결과의 관계를 그래프로 분석해 테스트
    • 오류 예측(Error Guessing): 경험을 바탕으로 오류 가능성이 높은 값을 테스트
    • 비교(Comparison): 여러 버전이나 시스템의 결과를 비교해 테스트
  • V-모델:
    • 요구사항 분석: 인수 테스트와 대응
    • 시스템 설계: 시스템 테스트와 대응
    • 아키텍처 설계: 통합 테스트와 대응
    • 모듈 설계: 단위 테스트와 대응
    • 구현: 설계 결과를 코드로 작성
  • 테스트 오라클:
    • 참 오라클: 모든 입력에 대해 기대 결과를 정확히 제공
    • 샘플링 오라클: 일부 입력값에 대해서만 기대 결과를 제공
    • 휴리스틱 오라클: 경험적 판단으로 기대 결과를 추정
    • 일관성 검사 오라클: 변경 전후 결과의 일관성을 확인
  • 애플리케이션 성능 측정:
    • 처리량: 단위 시간 동안 처리하는 작업량
    • 응답 시간: 요청 후 첫 응답이 오기까지 걸리는 시간
    • 경과 시간: 요청부터 완료까지 전체 처리에 걸리는 시간
    • 자원 사용률: CPU, 메모리, 네트워크 등 자원 사용 정도

8. SQL 응용 ​

  • 관계 해석:
    • 관계 데이터의 연산을 표기하는 방법. 수학의 술어 해석에 기반한다.
    • 튜플 관계 해석, 도메인 관계 해석으로 나눌 수 있음.
  • 관계 대수
    • 선택(Selection):
      • σ: 조건을 만족하는 튜플, 즉 행을 선택하는 연산
      • σ학과='컴퓨터공학과'(학생)
    • 프로젝트(Projection):
      • π: 릴레이션에서 특정 속성, 즉 열만 추출하는 연산
      • π학번, 이름(학생)
    • 합집합(Union):
      • ∪: 두 릴레이션에 존재하는 모든 튜플을 합치는 연산
      • R ∪ S
    • 교집합(Intersection):
      • ∩: 두 릴레이션에 공통으로 존재하는 튜플만 추출하는 연산
      • R ∩ S
    • 차집합(Difference):
      • −: 첫 번째 릴레이션에는 존재하지만 두 번째 릴레이션에는 존재하지 않는 튜플을 추출하는 연산
      • R − S
    • 카티션 프로덕트(Cartesian Product):
      • ×: 두 릴레이션의 모든 튜플 조합을 만드는 연산
      • R × S
    • 조인(Join):
      • ⋈: 두 릴레이션에서 관련 있는 튜플을 조건에 따라 결합하는 연산
      • 학생 ⋈ 학생.학과번호 = 학과.학과번호 학과
    • 디비전(Division):
      • ÷: 두 릴레이션 A, B에 대해 B 릴레이션의 모든 조건을 만족하는 A의 튜플을 구하는 연산
      • 수강 ÷ 과목
    • 세타 조인(Theta Join):
      • ⋈θ: =, <, >, ≤, ≥, ≠ 등의 비교 연산자를 조건으로 사용하는 조인 연산
      • R ⋈R.A < S.B S
    • 동등 조인(Equi Join):
      • ⋈=: 조인 조건이 등호(=)인 조인 연산
      • 학생 ⋈학생.학과번호 = 학과.학과번호 학과
    • 자연 조인(Natural Join):
      • ⋈: 두 릴레이션에서 같은 이름의 속성을 기준으로 자동으로 조인하고 중복 속성은 제거하는 연산
      • 학생 ⋈ 학과
    • 왼쪽 외부 조인(Left Outer Join):
      • ⟕: 왼쪽 릴레이션의 모든 튜플을 보존하고, 오른쪽 릴레이션과 매칭되지 않는 값은 NULL로 채우는 연산
      • 학생 ⟕ 학과
    • 오른쪽 외부 조인(Right Outer Join):
      • ⟖: 오른쪽 릴레이션의 모든 튜플을 보존하고, 왼쪽 릴레이션과 매칭되지 않는 값은 NULL로 채우는 연산
      • 학생 ⟖ 학과
    • 완전 외부 조인(Full Outer Join):
      • ⟗: 양쪽 릴레이션의 모든 튜플을 보존하고, 매칭되지 않는 값은 NULL로 채우는 연산
      • 학생 ⟗ 학과
  • 트랜잭션:
    • 데이터베이스를 조작하는 논리적 연산들이 하나 이상 모인 단위 작업이다.
    • 특징: 원자성, 일관성, 고립성, 지속성
  • 데이터 제어어(DCL)
    • 트랜잭션 복구 관련 용어
      • LOG
        • DBMS가 데이터베이스의 처리 내용, 이용 상황, 상태 변화 등을 시간 순서대로 기록한 것
        • 장애 발생 시 복구 작업의 근거가 된다.
      • REDO
        • 이미 완료된 트랜잭션의 작업을 다시 수행하는 연산
        • 트랜잭션의 start와 commit 기록이 모두 있는 경우 수행한다.
        • 로그를 이용하여 해당 데이터 항목을 이전 값에서 이후 값으로 변경한다.
        • 문제의 ①에 해당한다.
      • UNDO
        • 완료되지 않은 트랜잭션의 작업을 취소하는 연산
        • 트랜잭션의 start 기록은 있지만 commit 기록이 없는 경우 수행한다.
        • 로그를 이용하여 해당 데이터 항목을 이후 값에서 이전 값으로 되돌린다.
        • 문제의 ②에 해당한다.
      • COMMIT
        • 트랜잭션이 정상적으로 완료되었음을 확정하는 연산
        • COMMIT 이후의 결과는 데이터베이스에 영구 반영된다.
      • ROLLBACK
        • 트랜잭션 수행 중 오류가 발생했을 때 변경 내용을 취소하고 이전 상태로 되돌리는 연산
        • 트랜잭션의 원자성을 보장하기 위해 사용된다.
      • RECOVERY
        • 장애 발생 시 로그, 백업 등을 이용하여 데이터베이스를 정상 상태로 복구하는 작업
        • REDO와 UNDO가 대표적인 복구 연산이다.
      • BACKUP
        • 장애나 데이터 손실에 대비하여 데이터베이스의 사본을 별도로 저장하는 작업
        • 복구 작업의 기반 자료가 된다.
      • CHECK
        • 데이터베이스의 상태나 무결성 조건을 검사하는 작업
        • 트랜잭션 복구 문제에서는 핵심 정답보다는 보기용 용어로 나오는 경우가 많다.

9. 소프트웨어 개발 보안 구축 ​

  • 데이터베이스 보안:
    • 무결성: 데이터가 정확하고 일관되게 유지되어야 함
    • 가용성: 인가된 사용자가 필요할 때 데이터에 접근할 수 있어야 함
    • 기밀성: 인가되지 않은 사용자에게 데이터가 노출되지 않아야 함
  • 접근 통제 정책
    • DAC: 임의적 접근 통제, 신분에 따른 통제
    • MAC: 강제적 접근 통제, 전문가에 의해 생성되는 규칙에 기반을 두는 통제
    • RBAC: 역할 기반 접근 통제, 사용자의 역할에 기반을 둔 통제
  • 암호화 알고리즘:
    • 대칭키 알고리즘: 암호화와 복호화에 같은 키를 사용하는 방식
      • DES: 56비트 키를 사용하는 오래된 블록 암호
      • SEED: 국내에서 개발된 128비트 블록 암호
      • AES: 현재 널리 쓰이는 표준 대칭키 블록 암호
      • ARIA: 국내 표준으로 개발된 블록 암호
      • IDEA: 128비트 키를 사용하는 블록 암호
      • Skipjack: 미국 NSA가 개발한 블록 암호
      • RC5: 가변 블록 크기와 키 길이를 지원하는 블록 암호
      • WEP: 무선 LAN 보안을 위한 초기 암호화 방식
      • TKIP: WEP의 취약점을 보완한 무선 LAN 보안 방식
    • 비대칭키 알고리즘: 공개키와 개인키를 사용하는 방식
      • RSA: 큰 수의 소인수분해 어려움에 기반한 공개키 암호
      • ECC: 타원곡선 문제에 기반한 공개키 암호
      • DSA: 전자서명 생성을 위한 공개키 알고리즘
    • 해시 암호: 임의 길이 데이터를 고정 길이 해시값으로 변환하는 방식
      • MD5: 128비트 해시값을 생성하지만 보안 취약점이 알려짐
      • SHA: SHA-1, SHA-2, SHA-3 등으로 구분되는 해시 알고리즘 계열
  • 보안 공격 유형
    • 웹 공격
      • SQL Injection: SQL 쿼리 삽입
      • XSS: 악성 스크립트 삽입
      • CSRF: 사용자의 인증 상태를 이용한 요청 위조
      • Directory Traversal: ../ 경로 조작
      • Command Injection: OS 명령어 삽입
    • 네트워크 공격
      • Sniffing: 패킷 도청
      • Spoofing: 주소/신분 위조
      • ARP Spoofing: IP-MAC 매핑 위조
      • DNS Spoofing: DNS 응답 위조
      • Session Hijacking: 세션 탈취
    • 서비스 거부 공격
      • DoS: 단일 출처 서비스 마비
      • DDoS: 분산 출처 서비스 마비
      • SYN Flooding: SYN 대량 전송
      • Smurf: ICMP + 브로드캐스트 + IP 위조
      • RUDY: 느린 HTTP POST
      • Slowloris: 느린 HTTP Header
    • 악성코드
      • Virus: 파일 감염
      • Worm: 자기 복제, 네트워크 전파
      • Trojan Horse: 정상 프로그램 위장
      • Ransomware: 파일 암호화 후 금전 요구
      • Spyware: 사용자 정보 수집
      • Rootkit: 관리자 권한 은닉
      • Key Logger: 키 입력 기록
    • 보안 프로토콜
      • SSL/TLS: 전송 계층 보안, HTTPS 기반
      • HTTPS: HTTP + SSL/TLS
      • S-HTTP: HTTP 메시지 단위 보안
      • IPsec: IP 계층 보안, VPN
      • SSH: 안전한 원격 접속
      • SFTP: SSH 기반 파일 전송
      • PGP/S-MIME: 이메일 암호화/전자서명
      • Kerberos: 티켓 기반 인증
      • RADIUS/TACACS+: AAA 인증
      • WPA/WPA2/TKIP: 무선 LAN 보안
      • DNSSEC: DNS 위조 방지
  • IPsec:
    • IP 계층에서 인증, 무결성, 기밀성을 제공하는 보안 프로토콜
    • VPN 구성에 자주 사용되며 AH, ESP, IKE 등의 구성 요소가 있음

11. 응용 SW 기초 기술 활용 ​

  • 프로세스 상태:
    • 프로세스가 생성되어 종료될 때까지 거치는 실행 단계
    • 제출, 접수, 준비, 실행, 대기, 종료 상태로 구분
    • 준비 상태는 CPU 할당을 기다리는 상태, 실행 상태는 CPU를 점유한 상태, 대기 상태는 입출력 등 이벤트 완료를 기다리는 상태
  • 프로세스 상태 전이:
    • Dispatch: 준비 상태의 프로세스가 CPU를 할당받아 실행 상태로 전이
    • Time Run Out: 할당 시간이 끝나 실행 상태에서 준비 상태로 전이
    • Block: 입출력 등 이벤트가 필요해 실행 상태에서 대기 상태로 전이
    • Wake Up: 입출력이 끝나 대기 상태에서 준비 상태로 전이
  • 스케줄링(SJF, HRN, SRN 등등):
    • 여러 프로세스에 CPU와 시스템 자원을 배분하는 작업
    • FCFS: 먼저 도착한 프로세스를 먼저 처리
    • SJF: 실행 시간이 가장 짧은 프로세스를 먼저 처리하는 비선점 방식
    • HRN: (대기시간 + 서비스 시간) / 서비스 시간으로 우선순위를 계산
    • SRN/SRT: 남은 실행 시간이 가장 짧은 프로세스를 우선 처리하는 선점 방식
    • RR: 정해진 시간만큼 실행한 뒤 다음 프로세스로 넘기는 선점 방식
  • IPv4, IPv6 (서브네팅):
    • IPv4: 8비트씩 4부분, 총 32비트 주소 체계
    • IPv6: 16비트씩 8부분, 총 128비트 주소 체계
    • 서브네팅: IP 주소의 호스트 ID 일부를 네트워크 ID로 사용해 네트워크를 더 작은 단위로 분할
    • IPv6 주소 유형: 유니캐스트, 멀티캐스트, 애니캐스트
  • OSI 7계층:
    • 네트워크 통신 기능을 7개 계층으로 나눈 표준 참조 모델
    • 응용 계층: 사용자 인터페이스 제공
    • 표현 계층: 인코딩, 암호화, 압축 등 데이터 표현 방식 처리
    • 세션 계층: 세션 연결, 유지, 종료 관리
    • 전송 계층: 종단 간 신뢰성 있는 데이터 전송
    • 네트워크 계층: IP 주소 기반 라우팅
    • 데이터 링크 계층: MAC 주소 기반 프레임 전송
    • 물리 계층: 비트 데이터를 전기/광/무선 신호로 전송
  • 프로토콜의 기본 요소 3가지 및 프로토콜의 의미:
    • 프로토콜: 서로 다른 시스템이 데이터를 교환하기 위해 정한 통신 규약
    • 구문: 데이터 형식, 구조, 순서, 길이
    • 의미: 각 필드의 의미와 제어 정보
    • 시간: 데이터 전송 순서, 속도, 타이밍
  • RAID:
    • 여러 디스크를 배열로 구성해 데이터를 분산 저장하거나 중복 저장하는 기술
    • RAID 0: 스트라이핑, 속도는 빠르지만 장애 복구가 어려움
    • RAID 1: 미러링, 같은 데이터를 다른 디스크에 복제
    • RAID 5: 패리티를 여러 디스크에 분산 저장
    • RAID 6: 패리티를 2개 저장해 RAID 5보다 장애 허용 범위가 넓음
  • 오류 대응 방안:
    • 이중화: 복수의 시스템을 활용하여 주 시스템에 장애가 발생하는 경우에도 정상적인 서비스가 제공될 수 있도록 하기 위한 기술
      • 액티브-액티브, 액티브-스탠바이로 나뉨.
    • 재해 복구 시스템(DRS):
      • Mirror Site: 주 센터와 동일한 수준으로 실시간 동기화해 즉시 복구 가능한 사이트
      • Hot Site: 주요 설비와 데이터를 갖추고 짧은 시간 안에 복구 가능한 사이트
      • Warm Site: 일부 설비를 갖추고 백업 데이터를 이용해 일정 시간 후 복구하는 사이트
      • Cold Site: 장소만 준비하고 장애 발생 후 장비와 데이터를 구성하는 사이트
    • 복구 목표 시간(RTO): 장애 발생 후 서비스가 복구되어야 하는 목표 시간
    • 복구 목표 시점(RPO): 장애 발생 시 손실을 허용할 수 있는 데이터의 기준 시점
    • 업무 영향 분석(BIA): 장애가 업무에 미치는 영향을 분석해 복구 우선순위를 정하는 작업
