SQL ​

중첩 서브쿼리와 JOIN ​

sql
SELECT COUNT(*)
FROM employee AS e
JOIN project AS p
  ON e.project_id = p.project_id
WHERE p.name IN (
    SELECT name
    FROM project p
    WHERE p.project_id IN (
        SELECT project_id
        FROM employee
        GROUP BY project_id
        HAVING COUNT(*) < 2
    )
);
  • 유형: 중첩 서브쿼리와 JOIN 결과를 안쪽 쿼리부터 추적하는 문제
  • 규칙: 서브쿼리는 가장 안쪽부터 계산하고, GROUP BY로 묶은 뒤 HAVING으로 그룹 조건을 검사한다.
  • 함정: project_id = 10은 project 테이블에 2행이 있어 JOIN 시 행이 증가하지만 최종 조건에서 모두 제외된다.
  • 예시: 직원 수가 2명 미만인 project_id는 20이고, 해당 프로젝트명은 Beta이다.
  • 메모: 복잡한 중첩 쿼리는 가장 안쪽 SELECT의 결과를 임시 테이블처럼 적어가며 바깥쪽으로 올라간다.

C static 지역변수 ​

c
#include <stdio.h>

int func() {
    static int x = 0;
    x += 2;
    return x;
}

int main() {
    int x = 1;
    int sum = 0;

    for (int i = 0; i < 4; i++) {
        x++;
        sum += func();
    }

    printf("%d", sum);

    return 0;
}
  • 유형: C의 static 지역변수가 함수 호출 사이에 값을 유지하는지 확인하는 문제
  • 규칙: 함수 내부 static 변수는 최초 한 번만 초기화되고 함수가 끝나도 값이 사라지지 않는다.
  • 함정: main의 x와 func의 x는 이름만 같을 뿐 서로 다른 변수이며, main의 x++는 출력에 영향을 주지 않는다.
  • 예시: func()의 반환값은 호출 순서대로 2, 4, 6, 8이다.
  • 메모: static 지역변수의 수명과 일반 지역변수의 차이를 묻는 문제이다.

Java 오버라이딩과 필드 숨김 ​

java
public class Main {
    public static void main(String[] args) {
        Base a = new Derivate();
        Derivate b = new Derivate();

        System.out.print(a.getX() + a.x + b.getX() + b.x);
    }
}

class Base {
    int x = 3;

    int getX() {
        return x * 2;
    }
}

class Derivate extends Base {
    int x = 7;

    @Override
    int getX() {
        return x * 3;
    }
}
  • 유형: Java 상속에서 오버라이딩된 메서드와 숨겨진 필드의 접근 기준을 구분하는 문제
  • 규칙: 인스턴스 메서드는 실제 객체 타입을 따르고, 필드는 참조 변수의 선언 타입을 따른다.
  • 함정: a는 실제로 Derivate 객체를 가리키지만 a.x는 Base의 x를 사용한다.
  • 예시: a.getX()는 Derivate.getX(), a.x는 Base.x가 선택된다.
  • 메모: 메서드는 실제 객체, 필드는 참조 타입으로 기억한다.

C 연결 리스트 값 교환 ​

c
#include <stdio.h>

struct Node {
    int value;
    struct Node* next;
};

void func(struct Node* node) {
    while (node != NULL && node->next != NULL) {
        int t = node->value;
        node->value = node->next->value;
        node->next->value = t;

        node = node->next->next;
    }
}

int main() {
    struct Node n1 = {1, NULL};
    struct Node n2 = {2, NULL};
    struct Node n3 = {3, NULL};

    n1.next = &n3;
    n3.next = &n2;

    func(&n1);

    struct Node* current = &n1;

    while (current != NULL) {
        printf("%d", current->value);
        current = current->next;
    }

    return 0;
}
  • 유형: 연결 리스트를 두 노드씩 건너뛰며 값만 교환하는 문제
  • 규칙: node = node->next->next로 현재 노드에서 두 칸 뒤의 노드로 이동한다.
  • 함정: 노드의 연결 순서를 바꾸는 것이 아니라 인접한 두 노드의 value만 교환한다.
  • 예시: n1(1) -> n3(3)에서 값이 교환되어 n1(3) -> n3(1)이 된다.
  • 메모: 첫 반복 후 node가 마지막 노드 n2를 가리키며 n2.next가 NULL이므로 반복이 종료된다.

Java 예외 처리 ​

java
public class ExceptionHandling {
    public static void main(String[] args) {
        int sum = 0;

        try {
            func();
        } catch (NullPointerException e) {
            sum = sum + 1;
        } catch (Exception e) {
            sum = sum + 10;
        } finally {
            sum = sum + 100;
        }

        System.out.print(sum);
    }

    static void func() throws Exception {
        throw new NullPointerException();
    }
}
  • 유형: Java 예외 처리에서 catch 선택 순서와 finally 실행 여부를 추적하는 문제
  • 규칙: 발생한 예외와 가장 먼저 일치하는 catch 하나만 실행되고, finally는 예외 처리 후에도 항상 실행된다.
  • 함정: NullPointerException은 Exception의 하위 클래스지만 앞의 전용 catch에서 이미 처리되므로 catch(Exception)은 실행되지 않는다.
  • 예시: NullPointerException 발생 → sum에 1 추가 → finally에서 100 추가 → 101
  • 메모: 구체적인 예외를 먼저 catch하고 상위 예외 Exception은 뒤에 배치해야 한다.

Java 제네릭과 오버로딩 ​

java
class Main {

    public static class Collection<T> {
        T value;

        public Collection(T t) {
            value = t;
        }

        public void print() {
            new Printer().print(value);
        }

        class Printer {
            void print(Integer a) {
                System.out.print("A" + a);
            }

            void print(Object a) {
                System.out.print("B" + a);
            }

            void print(Number a) {
                System.out.print("C" + a);
            }
        }
    }

    public static void main(String[] args) {
        new Collection<>(0).print();
    }
}
  • 유형: Java 제네릭 타입 변수와 메서드 오버로딩의 컴파일 시점 선택 문제
  • 규칙: 오버로딩은 실행 시점의 실제 객체가 아니라 호출식에 나타난 변수의 선언 타입을 기준으로 결정한다.
  • 함정: new Collection<>(0)에서 T는 Integer로 추론되지만, 제네릭 클래스 내부의 value는 컴파일할 때 타입 변수 T로 판단하며 T의 기본 상한은 Object이다.
  • 예시: print(value)는 print(Object)를 호출하지만 print((Integer) value)는 print(Integer)를 호출한다.
  • 메모: 실제 값이 Integer라는 이유로 print(Integer)가 선택되지 않으며, 오버로딩은 컴파일 시점 타입을 본다.
